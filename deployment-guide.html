Section 01

Prerequisites

Before deploying the ABN AMRO Developer Tools, ensure the following requirements are met in your environment.

Requirement Details
Salesforce CLI The sf CLI (v2+) must be installed and authenticated to your target org. Install from npm install -g @salesforce/cli or the Salesforce CLI installer.
Org Access You must have access to the target org with the System Administrator profile, or a profile with deploy permissions and Modify All Data.
Project Structure A valid Salesforce DX project structure with sfdx-project.json at the root. The force-app/main/default directory must contain the classes and LWC source.
API Version API version 65.0 or higher. Earlier versions may lack support for required metadata types or LWC features.
Remote Site Setting The SelfOrgCalloutHelper class requires the org to allow callouts to itself. A Remote Site Setting pointing to the org's own instance URL must exist (used for REST cursor pagination).
ℹ Info
Verify your CLI version with sf version and your org authentication with sf org list. If your target org is not listed, authenticate with sf org login web --alias <your-org-alias>.
Section 02

Package Contents

The Developer Tools package includes the following metadata components.

Type Name Description
Apex DevToolsAccessController Permission checks and access gate logic
Apex ApexRunnerController Anonymous Apex execution and debug log retrieval
Apex MetadataBrowserController EntityDefinition-based object listing, field describes, and object detail
Apex DataLoaderController Bulk insert, update, upsert, and delete operations
Apex SOQLExplorerController SOQL query execution, cursor pagination, and inline record editing
Apex RecordInspectorController Record detail retrieval, field-level security, and record updates
Apex SecurityUtil Shared utility for global describe caching and permission checks
Apex SelfOrgCalloutHelper REST API callouts to the same org (cursor-based query pagination)
LWC devTools Single Lightning Web Component (12 files) containing all four tools: Record Inspector, SOQL Explorer, Apex Runner, and Data Loader
Permission Dev_Tools_Access Custom permission granting base access (Record Inspector)
Permission Dev_Tools_Power Custom permission granting elevated access (SOQL Explorer, Apex Runner, Data Loader)
Permission ABN_Dev_Tools_Admin Permission set bundling both custom permissions for full admin access
Config Custom Application + FlexiPage Optional. Pre-built Lightning app and page. See warning below.
⚠ Warning
The Custom Application and FlexiPage XML files may contain pre-existing XML issues that cause deployment failures. It is recommended to skip these components and manually configure the utility bar instead (see Section 5).
Section 03

Deployment Steps

Follow these steps to deploy the Developer Tools to your target Salesforce org.

1

Obtain the project source

Clone or download the project repository to your local machine. Ensure the sfdx-project.json file is present at the project root.

# Clone the repository (or extract the archive)
git clone <repository-url> abn-inspector
cd abn-inspector
2

Authenticate to the target org

If you have not already authenticated, log in via the browser flow.

sf org login web --alias <your-org-alias> --instance-url https://<instance>.salesforce.com
3

Deploy Apex classes and the LWC component

Deploy only the classes and the devTools LWC. This avoids the FlexiPage and CustomApplication XML issues.

sf project deploy start \
  --source-dir force-app/main/default/classes \
  --source-dir force-app/main/default/lwc/devTools \
  --target-org <your-org-alias> \
  --wait 10
⚠ Warning
Do not deploy the entire force-app directory. The FlexiPage and CustomApplication metadata may fail due to pre-existing XML issues. Deploy only the classes and lwc/devTools directories as shown above.
4

Deploy custom permissions and permission set

If the custom permissions and permission set are stored in separate directories, deploy them as well.

sf project deploy start \
  --source-dir force-app/main/default/customPermissions \
  --source-dir force-app/main/default/permissionsets \
  --target-org <your-org-alias> \
  --wait 10
5

Verify deployment success

Check the most recent deployment status to confirm all components deployed without errors.

sf project deploy report --use-most-recent
✓ Success
You should see "Status: Succeeded" with all components listed as deployed. If any component fails, review the error messages and refer to the Troubleshooting section.
Section 04

Permission Setup

The Developer Tools use a two-tier permission model controlled by custom permissions. Users must be assigned the appropriate permission set to access the tools.

Assign the Permission Set

Assign the ABN_Dev_Tools_Admin permission set to grant full access to all four tools.

sf org assign permset \
  --name ABN_Dev_Tools_Admin \
  --target-org <your-org-alias>

To assign to a specific user other than yourself, add the --on-behalf-of flag:

sf org assign permset \
  --name ABN_Dev_Tools_Admin \
  --on-behalf-of user@example.com \
  --target-org <your-org-alias>

Permission Model

Permission Access Level Tools Enabled
Dev_Tools_Access Base access Record Inspector
Dev_Tools_Power Elevated access SOQL Explorer, Apex Runner, Data Loader
ABN_Dev_Tools_Admin Full access (bundles both) All four tools
ℹ Info
You can create custom permission sets that include only Dev_Tools_Access for users who should have read-only Record Inspector access without the elevated SOQL, Apex, and Data Loader capabilities.
Section 05

Adding to Utility Bar

The Developer Tools component is designed to run as a utility bar item within any Lightning App. Follow these steps to add it.

1

Open the App Builder

Navigate to Setup → search for "App Manager" → find your target Lightning App → click Edit.

2

Navigate to Utility Items

In the App Settings sidebar, click "Utility Items (Desktop Only)".

3

Add the component

Click "Add Utility Item" and search for devTools. Select the c:devTools Lightning Web Component.

4

Configure the panel

Set the following properties:

Property Recommended Value
Label Developer Tools
Icon utility:setup or utility:inspector_panel
Panel Width 480 (px)
Panel Height 600 (px)
Start Automatically Unchecked
5

Save and activate

Click Save. If prompted, activate the app for the appropriate profiles. The Developer Tools icon will now appear in the utility bar at the bottom of the screen.

Section 06

Post-Installation Verification

After deployment and configuration, run through the following checklist to confirm everything is working correctly.

  • Component loads in utility bar — Click the Developer Tools icon in the utility bar. The panel should open and display the tool navigation tabs.
  • Access control works — As a user without the permission set, the component should display an "Access Restricted" message and block all functionality.
  • Record Inspector — Paste any valid 15- or 18-character Record ID. The inspector should load the record's fields, values, and metadata.
  • SOQL Explorer — Run a simple query: 
    SELECT Id, Name FROM Account LIMIT 5
    Results should appear in a tabular format with column headers.
  • Apex Runner — Execute a basic anonymous Apex snippet: 
    System.debug('Hello World');
    The debug log output should appear in the results panel.
  • Data Loader — Navigate to the Data Loader tab. The interface should be visible and functional (object selection, CSV upload area, operation selector).
✓ All checks passed
If all items above are confirmed, the deployment is complete. The ABN AMRO Developer Tools are ready for use.
Section 07

Troubleshooting

Common issues encountered during or after deployment, with their solutions.

Problem Solution
Deployment fails with FlexiPage or CustomApplication errors Do not deploy the full force-app directory. Deploy only the classes and lwc/devTools subdirectories as described in Section 3. These XML files have known pre-existing issues.
"Access Restricted" shown for permissioned user Verify the permission set assignment: sf org assign permset --name ABN_Dev_Tools_Admin --target-org <alias>. Ensure both Dev_Tools_Access and Dev_Tools_Power custom permissions are included in the set. The user may need to log out and log back in.
Component not appearing in utility bar Confirm the Lightning App was saved and activated after adding the utility item. Check that the user's profile is assigned to the app. Clear the browser cache or try an incognito window.
SOQL queries fail or return no results Ensure the running user has object-level and field-level read permissions for the queried object. Check that the API version in sfdx-project.json matches the deployed version.
Apex Runner returns "Compile Error" This indicates a syntax error in the anonymous Apex code, not a deployment issue. Review the code and try again. The error message will include the line and column number.
Cursor pagination / REST queries fail The SelfOrgCalloutHelper class makes HTTP callouts to the org's own REST API. Ensure a Remote Site Setting exists for your org's instance URL (e.g., https://<instance>.my.salesforce.com). Also check that the running user has API Enabled permission.
Data Loader operations timeout Large data volumes may exceed governor limits. Reduce batch sizes or split CSV files into smaller chunks. The Data Loader processes records in batches to stay within Apex limits.
Debug log not appearing in Apex Runner The tool creates a trace flag for the running user. If trace flags are at capacity (max 20 per org), delete old trace flags from Setup → Debug Logs. The tool caches the trace flag once per session.
sf org login web fails or times out Ensure you are on a network that allows browser redirects to Salesforce. Try specifying the instance URL explicitly with --instance-url. For sandboxes, use https://test.salesforce.com.
ℹ Need help?
If you encounter an issue not listed above, check the browser developer console (F12) for JavaScript errors, and the Salesforce debug logs for Apex exceptions. Both sources will provide detailed error context.
