1. Architecture Overview

ABN AMRO Developer Tools is built as a single Lightning Web Component (devTools) that bundles four developer tools into one utility bar panel: Record Inspector, SOQL Explorer, Apex Runner, and Data Loader.

Component Architecture

The application follows a monolithic LWC pattern where a single component houses all tool views. The main controller (devTools.js, approximately 2200 lines) coordinates between 8 Apex controllers, 1 utility class, and 7 JavaScript helper modules.

2. Deployment Guide

Two ways to deploy BALCOL Inspector to any Salesforce org — one-click or CLI.

Option 1: One-Click Deploy

Deploy directly from the GitHub repository — no CLI required. Click the button below, authenticate to your target org, and the deployment runs automatically.

Option 2: CLI Deploy

Prerequisites

  • Salesforce CLI v2.x or later installed
  • System Administrator profile (or equivalent deploy permissions)
  • Target org authenticated via sf org login web

Deploy Command

sf project deploy start \
  --source-dir force-app/main/default/classes \
  --source-dir force-app/main/default/lwc/devTools \
  --target-org myorg \
  --wait 10

Permission Set

The deployment includes a single permission set that grants full access:

Permission SetIncludesAssign To
ABN Dev Tools Admin Dev_Tools_Access + Dev_Tools_Power custom permissions, all Apex class access, VF page access Developers, administrators, power users
Granular Access
For read-only access (Record Inspector + SOQL Explorer only), create a second permission set with just the Dev_Tools_Access custom permission. The code checks two custom permissions internally:
  • Dev_Tools_Access — Record Inspector, SOQL Explorer
  • Dev_Tools_Power — Apex Runner, Data Loader

Assignment Steps

  1. Navigate to Setup → Permission Sets
  2. Click on ABN Dev Tools Admin
  3. Click Manage Assignments → Add Assignment
  4. Select users and click Assign

Utility Bar Setup

  1. Navigate to Setup → App Manager
  2. Click Edit on the target Lightning App
  3. Go to Utility Items (Desktop Only)
  4. Click Add Utility Item and select the devTools component
  5. Configure the panel:
    • Label: "BALCOL Inspector" or "Developer Tools"
    • Panel Width: 480px
    • Panel Height: 600px
    • Start Automatically: Checked
  6. Click Save

Post-Install Checklist

  • Verify Remote Site Settings for self-org callouts (Tooling API, REST API)
  • Check Visualforce page DevToolsSessionId exists (session ID retrieval)
  • Confirm Named Credential or Connected App if needed for REST callouts
  • Assign ABN Dev Tools Admin permission set to test user
  • Test access with the permission set assigned — verify all 4 tools are available
  • Test access with no permission — verify access denied message appears
  • Verify SOQL query execution against a known object (e.g., SELECT Id FROM Account LIMIT 1)
  • Verify Apex Runner can execute anonymous code and retrieve debug logs

4.3 Helper Modules

soqlAutocomplete.js

Manages schema metadata caching and provides intelligent SOQL autocomplete suggestions.

  • SchemaCache — Singleton that caches field metadata per object, shared across all query tabs
  • Relationship traversal — Supports up to 4 levels deep (e.g., Account.Owner.Manager.Name)
  • _pendingFetches — Deduplicates concurrent schema requests for the same object
  • 50ms debounce — Reduced from 200ms for responsive autocomplete
  • Monotonic request ID — Guards against stale closure responses

soqlHighlighter.js

Generates syntax-highlighted HTML for the SOQL editor overlay.

  • Keyword highlighting (SELECT, FROM, WHERE, etc.)
  • String literal highlighting
  • Function highlighting (COUNT, MAX, etc.)
  • Overlay textarea pattern — transparent textarea overlays a styled <code> element with lwc:dom="manual"

soqlTokenizer.js

Tokenizes SOQL input and provides context-aware clause information.

  • getClauseContext() — Returns current clause (SELECT, WHERE, ORDER BY, etc.), relationship path, and expected next token type
  • Used by autocomplete to determine whether to suggest fields, objects, operators, or functions

csvExporter.js

Multi-format export utility for query results.

  • CSV — RFC 4180 compliant with UTF-8 BOM for Excel compatibility
  • JSON — Pretty-printed array of objects
  • Excel XML — SpreadsheetML format for direct Excel opening

csvParser.js

Parses CSV files uploaded to the Data Loader.

  • Handles quoted fields, embedded commas, and newlines
  • Auto-maps column headers to Salesforce field API names

historyManager.js

Manages SOQL query history using localStorage.

  • Stores up to 20 recent queries with abn_devtools_* prefix
  • Supports named saved queries (persist across sessions)

preferencesManager.js

User preference management with localStorage.

  • Theme selection, confirm destructive actions toggle
  • Default record count, preferred export format
  • All keys prefixed with abn_devtools_

devToolsConstants.js

Centralized constants and configuration.

  • MODULES — Tool definitions with labels, icons, and permission requirements
  • FIELD_TYPE_ICONS — Maps Salesforce field types to SLDS icon names
  • APEX_SNIPPETS — Code snippets for the Apex Runner quick-insert menu
  • EDITABLE_COLUMN_TYPES — Set of field types that support inline editing

4.5 Performance Optimizations

The component has been through three phases of performance optimization:

Phase 1: JavaScript Caching

  • Preferences caching — Reads from localStorage once, avoids repeated parsing
  • SchemaCache sharing — Single cache instance shared across all query tabs
  • Debounce reduction — Autocomplete debounce reduced from 200ms to 50ms
  • Getter memoization — _riFilteredFields (cache key), _currentTabColumns (ref check), _apexDebugLines (source check)

Phase 2: Apex Caching

  • DataLoader fieldMap — Pre-cached DescribeFieldResult map for type casting
  • SecurityUtil globalDescribe — Cached once per transaction via static variable
  • Child relationships — Capped at 20 with LIMIT 1 existence check (shows "has records" / "empty")
  • Trace flag — ensureTraceFlag called once per session, not per execution
  • Pattern.compile — Moved to static class constant (ApexRunnerController DEBUG_PATTERN)

Phase 3: Progressive Batch Loading

  • REST cursor pagination — executeQueryCursor replaces OFFSET-based queries
  • No 2000 offset limit — Uses nextRecordsUrl from REST API response
  • Auto-streaming — Client-side while loop fetches 200 records per batch until complete
  • Sforce-Query-Options header — Controls batch size at REST API level
Design Decision
Phase 2.4 (lazy picklist loading) was evaluated and skipped. Picklist values are part of the same DescribeFieldResult call, so separating them would add latency without reducing payload.

5.3 Test Patterns & Techniques

HttpCalloutMock

Controllers that make HTTP callouts (ApexRunnerController, SOQLExplorerController cursor/plan methods) use inner classes implementing HttpCalloutMock to simulate Tooling API and REST API responses.

SecurityUtil.testPermissionOverride

A @TestVisible private static Boolean field that bypasses FeatureManagement.checkPermission in test context. This allows test classes to run without requiring custom permission assignment, which is not possible in standard test setup.

AuraHandledException Handling

In test context, AuraHandledException.getMessage() returns "Script-thrown exception" regardless of the original message. Test methods catch Exception (the parent class) and verify that an exception was thrown without asserting on the message content.

Test Data Isolation

Each test class uses @TestSetup methods to create isolated test data. Queries use ORDER BY CreatedDate DESC LIMIT 1 instead of WHERE Name = ... to avoid field filterability issues in certain org configurations.

5.4 Coverage Justification

The weighted average code coverage across all 8 controllers is 89%. While 100% is not achieved, the remaining uncovered lines are architecturally justified and do not represent untested business logic:

  • SelfOrgCalloutHelper (72%) — Lines 13–18 retrieve the session ID through a Visualforce page at runtime. The code contains an explicit Test.isRunningTest() guard that returns UserInfo.getSessionId() directly in test context, making the VF page path unreachable by design. This is a standard Salesforce pattern for session-based callouts.
  • Defensive catch blocks — Several controllers contain nested catch (SecurityUtil.SecurityException) and catch (Exception) blocks that re-throw as AuraHandledException. In test context with an admin user, these security exceptions cannot be triggered because the user has full CRUD access to all standard objects. These catch blocks are essential for production safety but untestable without custom objects with restricted permissions.
  • RecordInspectorController line 65 — The “no accessible fields” guard throws when a user has object-level READ but zero field-level access. This scenario is impossible for standard objects like Account in test context.
  • ApexRunnerController trace flag branches — The ensureTraceFlag() method has multiple HTTP-based branches covered by 5 dedicated mock classes. The remaining uncovered lines (168–171, 237) are in response-parsing blocks for edge cases in the Tooling API response format.
Coverage Strategy
All uncovered lines are defensive error handlers, platform-specific guards (Test.isRunningTest()), or permission boundaries that cannot be exercised in standard test contexts. The 89% average reflects thorough testing of all reachable business logic, with the remaining 11% being infrastructure-level safety nets.

7. Security Model

The Developer Tools implement a two-tier permission model with comprehensive CRUD/FLS enforcement and SOQL injection prevention.

7.1 Permission Architecture

LayerMechanismEnforced By
Visibility Dev_Tools_Access custom permission DevToolsAccessController.hasAccess() — component hides if false
Feature Gating Dev_Tools_Power custom permission SecurityUtil.hasPowerToolsPermission() — checked in every power method
Object CRUD DescribeSObjectResult checks SecurityUtil.checkObjectCRUD() — throws SecurityException
Field-Level Security DescribeFieldResult checks + Security.stripInaccessible SecurityUtil.getAccessibleFields() and stripInaccessible in DataLoaderController
Sharing Rules with sharing keyword All controllers declared with sharing

7.2 CRUD/FLS Enforcement

Every Apex method that performs DML or queries user data follows this enforcement pattern:

  1. Permission check — SecurityUtil.hasPowerToolsPermission() for power features
  2. Object CRUD check — SecurityUtil.checkObjectCRUD(objType, operation) throws if denied
  3. Field-level filtering — SecurityUtil.getAccessibleFields() or Security.stripInaccessible()
  4. Execute operation — DML or SOQL with only accessible fields
// Example enforcement pattern in RecordInspectorController
SecurityUtil.checkObjectCRUD(objType, CRUDOperation.READ);
List<String> accessibleFields = SecurityUtil.getAllAccessibleFields(
    objType, CRUDOperation.READ
);
String query = 'SELECT ' + String.join(accessibleFields, ',')
    + ' FROM ' + objectName + ' WHERE Id = :recordId';

7.3 SOQL Injection Prevention

TechniqueUsageControllers
Bind variables :recordId for all record ID references RecordInspectorController, SOQLExplorerController
String.escapeSingleQuotes All user-provided string values in dynamic SOQL SecurityUtil.sanitizeForSOQL()
Schema.describe-derived names Object and field names from getGlobalDescribe() / getDescribe() RecordInspectorController (child rels, history)
API name validation Regex validation before use in queries SecurityUtil.validateApiName()
Design Decision: User-Provided SOQL
The SOQL Explorer tool accepts and executes user-provided SOQL queries by design. This is an intentional feature for developer use, gated behind the Dev_Tools_Power permission. Only users explicitly assigned this permission can execute arbitrary queries, and all queries run with sharing to enforce record-level access.

Roadmap & Beta Status

Beta Release
This release is intended for internal developer use within the ABN AMRO Salesforce team. While all features are functional and security-tested, the tool has not yet undergone extended production usage at scale. Users should report any issues or feature requests through the standard team channels.

Current Focus: Stability

The primary focus for the current release cycle is stability and reliability:

  • Monitor LWC component performance under real-world usage patterns (large field counts, complex queries, concurrent tabs)
  • Validate Apex controller behavior across different org configurations and permission sets
  • Ensure consistent behavior across Salesforce API version upgrades
  • Collect user feedback on UX patterns and workflow ergonomics

Planned Improvements

Future enhancements will be prioritized based on user feedback and usage data:

  • SOQL Editor — Query history persistence, saved query library, export to multiple formats
  • Record Inspector — Field-level comparison between records, record cloning, related record navigation
  • Apex Runner — Code snippet library with org-level sharing, execution history with log correlation
  • Data Loader — Larger batch support via chunked processing, mapping template saves, rollback support
  • General — Keyboard shortcuts, user preferences sync, dark/light theme toggle

Feedback & Contribution

Performance and functionality will be continuously reviewed based on user input. If you encounter bugs, performance issues, or have feature requests, please reach out through the Salesforce Development team channel. All feedback is tracked and prioritized for upcoming releases.
