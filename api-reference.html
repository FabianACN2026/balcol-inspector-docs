Security Model

All controllers in the Developer Tools suite enforce security through SecurityUtil, a centralized utility class that provides consistent CRUD, FLS, and permission checks. Every controller class is declared with sharing to enforce record-level sharing rules.

Layer 1: Custom Permission Gate

Access to the tool suite is gated by the Dev_Tools_Access custom permission, checked via FeatureManagement.checkPermission() in DevToolsAccessController.hasAccess(). The LWC checks this on load and hides all UI if denied.

Layer 2: Power Tools Permission

Elevated operations (Apex execution, SOQL queries, DML, exports) require the Dev_Tools_Power custom permission, checked via SecurityUtil.hasPowerToolsPermission(). Methods throw AuraHandledException if denied. ensureTraceFlag() silently returns instead of throwing (non-critical setup).

Layer 3: Object-Level CRUD

SecurityUtil.checkObjectCRUD(objectType, operation) validates the user has the required CRUD permission (READ, CREATEABLE, UPDATEABLE, DELETEABLE) on the target SObject. Throws SecurityUtil.SecurityException if denied. Object names are validated against Schema.getGlobalDescribe() (cached per transaction) via validateObjectName().

Layer 4: Field-Level Security (FLS)

Two complementary mechanisms enforce FLS:

Pre-query filtering: SecurityUtil.getAccessibleFields() and getAllAccessibleFields() filter field lists to only those the user can access for the requested operation.

Post-query stripping: Security.stripInaccessible() (platform API) removes inaccessible field values from query results and DML payloads, supporting READABLE, CREATABLE, UPDATABLE, and UPSERTABLE access types.

Layer 5: Input Validation

SecurityUtil.validateObjectName() ensures object names exist in the global describe, preventing injection of arbitrary object names. SecurityUtil.validateApiName() enforces a ^[a-zA-Z][a-zA-Z0-9_]*$ regex pattern for API names. SecurityUtil.sanitizeForSOQL() applies String.escapeSingleQuotes() for safe SOQL construction.

Session-Based API Authentication

SelfOrgCalloutHelper authenticates REST and Tooling API callouts using the current user's session Id (obtained from a Visualforce page or UserInfo.getSessionId()). All requests are scoped to the same org (URL.getOrgDomainUrl()) and inherit the user's permissions. The session Id is cached per-request to avoid redundant VF page evaluations.

Permission Set Assignment Users need the Dev_Tools_Access permission set for basic access and the Dev_Tools_Power permission set for data operations. Both are custom permissions checked at runtime — they are not tied to profiles.
